    Core functionality

    accept_mutex

    如果启用了 accept_mutex，工作进程将依次接受新连接。否则，所有的工作进程都会收到新连接的通知，如果新连接的数量很少，一些工作进程可能会浪费系统资源。

    Syntax: accept_mutex on | off;
Default: accept_mutex off;
Context: events

    user

    定义 worker 进程使用的凭据，即用户和组。

    Syntax: user user [group];
Default: user nobody nobody;
Context: main

    worker_processes

    定义 worker 进程的数量，通常等于 CPU 核心数，也可以设置为 auto。

    Syntax: worker_processes number | auto;
Default: worker_processes 1;
Context: main

    pid

    定义一个用于存储 master 进程 ID 的文件。

    Syntax: pid file;
Default: pid logs/nginx.pid;
Context: main

    error_log

    定义错误日志的路径，级别。日志的错误级别大于等于 level 指定的值才会写入错误日志中。

    Syntax: error_log file [level];
Default: error_log logs/error.log error;
Context: main,http,mail,stream,server,location

    Level:  # debug|info|notice|warn|error|crit|alert|emerg

    debug 级别的日志需编译时使用 --with-debug 开启 debug 开关

    worker_rlimit_nofile

    指定 worker 进程最大打开的文件描述符数，一般设置为 echo "$(ulimit -n)/worker_processes" | bc。

    Syntax: worker_rlimit_nofile number;
Default: —
Context: main

    include

    在当前位置包含指定的配置文件。

    Syntax: include file | mask;
Default: —
Context: any

    use

    指定要使用的事件处理模型。一般不指定，nginx 默认使用最有效的方法。

    Syntax: use method;
Default: —
Context: events
Method: select | poll | kqueue | epoll | resig | /dev/poll | eventport

    worker_connections

    设置 worker 进程同时打开连接的最大数量，包括所有连接（如与代理服务器的连接等），不仅是与客户端的连接。另一个考虑因素是实际同时连接数不能超过 的当前限制打开文件的最大数量，可按 worker_rlimit_nofile 更改。

    Syntax: worker_connections number;
Default: worker_connections 512;
Context: events

    worker_cpu_affinity

    将 worker 绑定到 CPU 集。每个 CPU 集由允许的 CPU 掩码表示。应该为每个工作进程定义一个单独的集。默认情况下，工作进程不绑定到任何特定 CPU。

    Syntax: worker_cpu_affinity cpumask ...;
        worker_cpu_affinity auto [cpumask];
Default: —
Context: main

    multi_accept

    如果禁用了 multi_accept, worker 进程将一次接受一个新连接。否则，worker 进程将一次接受所有新连接。

    如果使用了 kqueue 连接处理方法，则忽略该指令，因为它报告了等待接受的新连接的数目。

    Syntax: multi_accept on | off;
Default: multi_accept off;
Context: events

    ngx_http_core_module

    alias

    替换访问的指定地址。

    Syntax: alias path;
Default: —
Context: location

    # Examples:

# 用户访问 www.zhums.com/image/1.png 时，实际查找的是 /data/nginx/static/image/1.png
location /image {
    alias /data/nginx/static/image/;
}

    注意：使用 alias 末尾一定要添加 /。 root 会将 location 定义的路径与 URI 叠加，而 alias 则只取定义路径。 

    # 如下两个 location 作用是相同的。
location /image {
    root /data/nginx/static;
}

location /image {
    alias /data/nginx/static/image/;
}

    default_type

    定义响应的默认 MIME 类型。可以用 types 指令设置文件名扩展到 MIME 类型的映射。

    Syntax: default_type mime-type;
Default: default_type text/plain;
Context: http,server,location

    error_page

    将指定错误码的请求转到指定 URI

    Syntax: error_page code ... [=[response]] uri;
Default: —
Context: http,server,location,location 中的 if
Response: 指定返回给客户端的状态码

    # Examples:

# 当客户端请求的状态码为 404 时会转到 /usr/share/nginx/html/404.html 页面，并设置返回给用户的状态码为 200
error_page 404 =200 /404.html
location = /404.html {
    root /usr/share/nginx/html;
}

    keepalive_requests

    设置通过一个长连接可以服务的最大请求数。达到最大请求数后，连接关闭。即 Nginx 和用户建立连接后，用户最大可通过此连接发送请求的个数。为了释放每个连接的内存分配，定期关闭连接是必要的。因此，使用过高的最大请求数可能会导致过多的内存使用，不建议使用。

    Syntax: keepalive_requests number;
Default: keepalive_requests 1000;
Context: http,server,location

    keepalive_time

    设置通过一个 keep-alive 连接可以处理请求的最大时间。到达这个时间之后，在后续的请求处理之后关闭连接。

    Syntax: keepalive_time time;

    location

    根据请求 URI 设置配置。

    Syntax: location [ = | ~ | ~* | ^~ ] uri { ... }
        location @name { ... }
Default: —
Context: server,location
    匹配模式

    • =: 精确匹配，匹配完全与 uri 一致的请求。

      location = /admin       # 匹配 uri 为 /admin 的请求

    • ^~: 前缀匹配，匹配以特定字符串开头的 uri。

      location ^~ /login      # 匹配 uri 以 /admin 开头的请求

    • ~: 正则匹配，表示 uri 包含正则表达式，且区分大小写。

    • ~*: 正则匹配，表示 uri 包含正则表达式，且不区分大小写。

    • '': 通用匹配，

    @name   # Nginx 内部跳转
    匹配模式的优先级
    1、=
2、^~
3、~ | ~*
4、普通前缀匹配，如 /static
5、/，通用
    匹配模式的匹配顺序
    • 优先级相同：正则看上下顺序，上面的优先；普通前缀匹配看长度，最长匹配的优先
    • 精确、前缀、正则、普通前缀都没有匹配到，最后再看通用匹配
    生产建议

    生产环境，一般至少有三个匹配规则定义

    第一个必选规则

    直接匹配网站根，通过域名访问网站首页比较频繁，使用这个会加速处理，比如说官网。

    可以是一个静态首页，也可以直接转发给后端应用服务器

    location = / {
    root   html;
    index  index.html index.htm;
}

    第二个必选规则

    是处理静态文件请求，这是 nginx 作为 http 服务器的强项，有两种配置模式，目录匹配或后缀匹配。

    location ^~ /static/ {
    root /webroot/static/;
}

location ~* \.(html|gif|jpg|jpeg|png|css|js|ico)$ {
    root /webroot/res/;
}

    第三个规则

    就是通用规则，比如用来转发带 .php、.jsp 后缀的动态请求到后端应用服务器。

    location / {
    proxy_pass http://tomcat_server;
}

    Examples:

    # Uri 以 /static/ 开头的请求，如果链接的状态为 404。则会匹配到 @index_error 这条规则上。
location /static/ {
    error_page 404 @index_error;
}
location @index_error {
    .....
}

location = / {}
# = 为精确匹配 /，主机名后面不能带任何字符串，比如访问 / 和 /data，则 / 匹配，/data 不匹配
# 再比如 location = /abc，则只匹配 /abc ，/abc/ 或 /abcd 不匹配。若 location /abc，则即匹配 /abc 、/abcd/ 同时也匹配 /abc/。

location / {}
# 因为所有的地址都以 / 开头，所以这条规则将匹配到所有请求 比如访问 / 和 /data, 则 / 匹配, /data 也匹配，但若后面是正则表达式会和最长字符串优先匹配（最长匹配）

location /documents/ {}
# 匹配任何以 /documents/ 开头的地址，匹配符合以后，还要继续往下搜索其它 location,只有其它 location 后面的正则表达式没有匹配到时，才会采用这一条

location ^~ /images/ {}
# 匹配任何以 /images/ 开头的地址，匹配符合以后，停止往下搜索正则，采用这一条

location ~* \.(gif|jpg|jpeg)$ {}
# 匹配所有以 gif、jpg 或 jpeg 结尾的请求然而，所有请求 /images/ 下的图片会被 location ^~ /images/ 处理，因为 ^~ 的优先级更高，所以到达不了这一条正则

location /images/abc {}
# 最长字符匹配到 /images/abc，优先级最低，继续往下搜索其它 location，会发现 ^~ 和 ~ 存在

location ~ /images/abc {}
# 匹配以/images/abc 开头的，优先级次之，只有去掉 location ^~ /images/ 才会采用这一条

    proxy_buffering

    启用或禁用代理服务器响应的缓冲。

    Syntax: proxy_buffering on | off;
Default: proxy_buffering on;
Context: http, server, location

    当开启缓冲时，Nginx 会先将来自后端服务器的响应数据缓存在本地，再返回给客户端。这可以提高响应速度和效率，尤其是在高并发场景下。

    缓存保存在 proxy_buffer_size 和 proxy_buffers 指令设置的缓冲区中。如果整个响应无法放入内存，则可以将其中一部分保存到磁盘上的 临时文件中。写入临时文件由proxy_max_temp_file_size 和 proxy_temp_file_write_size指令控制 。

    当禁用缓冲时，响应将在收到后立即同步传递到客户端。 nginx 不会尝试从代理服务器读取整个响应。 nginx 一次可以从服务器接收的数据的最大大小由proxy_buffer_size指令设置。

    还可以通过在 "X-Accel-Buffering" 响应头字段中传递 "yes" 或 "no" 来启用或禁用缓冲。可以使用 proxy_ignore_headers 指令禁用此功能。

    proxy_buffer_size

    设置 Nginx 用来存储来自后端服务器响应头部的缓冲区大小。默认情况下，缓冲区大小等于一页内存。如果响应头部的大小超过了这个缓冲区，剩余的部分会被存储在后面的缓冲区中。

    默认值：4k（32位系统）或8k（64位系统）

    Syntax: proxy_buffer_size size;
Default: proxy_buffer_size 4k|8k;
Context: http, server, location

    proxy_buffers

    Syntax: proxy_buffers number size;
Default: proxy_buffers 8 4k|8k;
Context: http, server, location

    upstream

    定义一组服务器。服务器可以监听不同的端口。

    Syntax: upstream name { ... }
Default: —
Context: http

    server

    定义服务器的地址和其他参数。地址可以是域名或 IP，如果没有指定端口，则使用端口 80。

    Syntax: server address [parameters];
Default: —
Context: upstream

    # parameters

weight=number       # 设置服务器的权重，默认为 1。
backup              # 将服务器标记为备份服务器。当主服务器不可用时，它将传递请求。不可与 hash, ip_hash, random 一起使用
down                # 将服务器标记为永久不可用。
