    注册存储库

    快照存储库是快照的集群外存储位置。必须先注册存储库，然后才能拍摄或恢复快照。

    先决条件

    • 要使用 Kibana 的快照和恢复功能，您必须拥有以下权限：
      • 集群权限：monitor、manage_slm、 cluster:admin/snapshot和cluster:admin/repository
      • 索引特权：all在monitor索引 上
    • 要注册快照存储库，集群的全局元数据必须是可写的。确保不存在任何阻止写入访问的 簇块。

    注意事项

    注册快照存储库时，请记住以下几点：

    • 每个快照存储库都是单独且独立的。 Elasticsearch 不在存储库之间共享数据。

    • 如果您向多个集群注册相同的快照存储库，则只有一个集群应该具有对该存储库的写访问权限。在其他集群上，将存储库注册为只读。

      这可以防止多个集群同时写入存储库并损坏存储库的内容。它还会阻止 Elasticsearch 缓存存储库的内容，这意味着其他集群所做的更改将立即可见。

    • 将 Elasticsearch 升级到较新版本时，您可以继续使用升级前使用的相同存储库。如果多个集群访问存储库，则它们都应该具有相同的版本。一旦存储库被特定版本的 Elasticsearch 修改，旧版本访问时它可能无法正常工作。

    管理快照存储库

    您可以通过两种方式注册和管理快照存储库：

    • Kibana 的快照和恢复功能
    • Elasticsearch 的快照存储库管理 API

    要在 Kibana 中管理仓库，请转到主菜单并单击堆栈 管理>快照和还原>存储库。要注册 snapshot repository，单击 Register repository。

    快照存储库类型

    Elasticsearch service 存储库类型

    自行管理的存储库类型

    如果您在自己的硬件上运行 Elasticsearch，则可以使用以下内置 快照存储库类型：

    • 共享文件系统存储库
    • 只读 URL 存储库
    • Source-only 存储库
    共享文件系统存储库

    要注册共享文件系统存储库，需要先将文件系统挂载到所有节点上的同一位置。然后将文件系统的路径或父目录添加到每个节点的 path.repo 设置中。正在运行的集群，需要 滚动重新启动 每个节点。

    默认情况下，网络文件系统 (NFS) 使用用户 ID (UID) 和组 ID (GID) 来跨节点匹配帐户。如果您的共享文件系统是 NFS 并且您的节点不使用相同的 UID 和 GID，请更新您的 NFS 配置以解决此问题。

    1、安装 NFS
    yum -y install nfs-utils rpcbind
systemctl enable --now rpcbind.service
systemctl enable --now nfs-server.service 
mkdir -p /data/es_data
echo "/data/es_data 10.0.0.0/24(rw)" >> /etc/exports
exportfs -rv        # 重新加载配置文件
    2、所有 ES 节点挂载共享目录
    mkdir /data/es_snapshot

showmount -e 10.0.0.17
mount -t nfs 10.0.0.17:/data/es_data /data/es_snapshot
echo "10.0.0.17:/data/es_data   /data/es_snapshot   nfs     defaults    0 0" >> /etc/fstab
    3、所有 ES 节点修改配置文件并重启
    echo 'path.repo: ["/data/es_snapshot"]' >> /etc/elasticsearch/elasticsearch.yml
systemctl restart elasticsearch
    4、注册存储库

    1、使用 Elasticsearch API 注册存储库

    PUT _snapshot/backups
{
  "type": "fs",
  "settings": {
    "location": "/data/es_snapshot",
  }
}

curl -X PUT "localhost:9200/_snapshot/backups?pretty" -H 'Content-Type: application/json' -d'
{
  "type": "fs",
  "settings": {
    "location": "/data/es_snapshot",
  }
}
'
    • type: 指定存储库类型
    • settings: 指定存储库的设置
    • location: 指定存储和检索快照的共享文件系统的位置。必须在集群中所有节点的配置文件中中注册 path.repo。

    如果 location 使用相对路径，Elasticsearch 将使用配置文件中 path.repo 的值解析路径。如下例子中的 location 即为 '/data/es_snapshot/backups-2'。

    PUT _snapshot/backups-2
{
  "type": "fs",
  "settings": {
    "location": "backups-2"        
  }
}

curl -X PUT "localhost:9200/_snapshot/backups-2?pretty" -H 'Content-Type: application/json' -d'
{
  "type": "fs",
  "settings": {
    "location": "backups-2"
  }
}
'

    POST _snapshot/backup/_verify

curl -X POST "localhost:9200/_snapshot/backup/_verify?pretty"

    删除存储库

    POST _snapshot/backup/_cleanup

curl -X POST "localhost:9200/_snapshot/backup/_cleanup?pretty"

    备份存储库

    创建快照

    先决条件

    • 要使用 Kibana 的快照和恢复功能，您必须拥有以下权限：
      • 集群权限：monitor、manage_slm、 cluster:admin/snapshot和cluster:admin/repository
      • 索引特权：all在monitor索引 上
    • 您只能从具有选定主节点 的正在运行的集群中获取快照 。
    • 快照存储库必须注册并可供集群使用。
    • 集群的全局元数据必须可读。要在快照中包含索引，索引及其元数据也必须可读。确保不存在任何阻止读取访问的 簇块或索引块。

    注意事项

    • 每个快照在其存储库中必须有唯一的名称。尝试创建与现有快照同名的快照将会失败。

    • 快照会自动进行重复数据删除。您可以频繁拍摄快照，而对存储开销的影响很小。

    • 每个快照在逻辑上都是独立的。您可以删除快照而不影响其他快照。

    • 拍摄快照可以暂时暂停分片分配。请参阅快照和分片分配。

    • 拍摄快照不会阻止索引或其他请求。但是，快照不会包含快照进程启动后所做的更改。

    • 您可以同时拍摄多个快照。集群 snapshot.max_concurrent_operations设置限制并发快照操作的最大数量。

    • 如果您在快照中包含数据流，则快照还包含该流的支持索引和元数据。

      您还可以在快照中仅包含特定的支持索引。但是，快照不会包含数据流的元数据或其其他支持索引。

    • 快照可以包含数据流，但排除特定的支持索引。当您恢复此类数据流时，它将仅包含快照中的后备索引。如果流的原始写入索引不在快照中，则快照中的最新后备索引将成为流的写入索引。

    使用 SLM 自动创建快照

    快照生命周期管理 (SLM) 是定期备份集群的最简单方法。 SLM 策略自动按照预设计划拍摄快照。该策略还可以根据您定义的保留规则删除快照。

    手动创建快照

    创建快照时，默认会备份所有打开的索引。如果您不想备份系统索引，例如以 .kibana、.security、.monitoring 等开头的索引，可在创建快照时指定需要备份的索引。

    PUT _snapshot/backup/snapshot-1?wait_for_completion=true

curl -X PUT "localhost:9200/_snapshot/my_repository/my_snapshot?wait_for_completion=true&pretty"

    wait_for_completion=true: 在快照任务完成后才返回响应，默认仅启动快照进程，该进程在后台运行。

    给指定索引创建快照

    PUT _snapshot/backup/snapshot-2
{
    "indices": "index_1,index_2",
    "wait_for_completion": "true"
}

curl -X PUT "localhost:9200/_snapshot/backup/snapshot-2?pretty" -H 'Content-Type: application/json' -d
{
  "indices": "index_1,index_2",
  "wait_for_completion": "true"
}

    监控快照

    监视任何当前正在运行的快照。

    GET _snapshot/backup/_current

curl -X GET "localhost:9200/_snapshot/backup/_current?pretty"

    获取参与当前正在运行的快照的每个分片的完整详细信息。

    GET _snapshot/_status

curl -X GET "localhost:9200/_snapshot/_status?pretty"

    删除快照

    DELETE _snapshot/backup/snapshot-1

curl -X DELETE "localhost:9200/_snapshot/backup/snapshot-1?pretty"

    还原快照

    快照是在集群外部存储数据副本的方法。可以在删除或硬件故障后恢复快照以恢复索引和数据流。还可以使用快照在集群之间传输数据。

    先决条件

    • 要使用 Kibana 的快照和恢复功能，您必须具有以下权限：

      • 集群权限：monitor、manage_slm、 cluster:admin/snapshot和cluster:admin/repository
      • 索引特权：all在monitor索引 上

    • 您只能将快照恢复到具有选定 主节点的正在运行的集群。快照的存储库必须已 注册并且可供集群使用。

    • 快照版本和集群版本必须兼容。请参阅 快照兼容性。

    • 要恢复快照，集群的全局元数据必须可写。确保不存在任何阻止写入的簇块。恢复操作忽略索引块。

    • 在恢复数据流之前，请确保集群包含 启用了数据流的匹配索引模板。要进行检查，请使用 Kibana 的索引管理功能或获取索引模板 API：

      GET _index_template/*?filter_path=index_templates.name,index_templates.index_template.index_patterns,index_templates.index_template.data_stream

      复制为卷曲尝试使用弹性

      如果不存在此类模板，您可以创建一个或 恢复包含该模板的集群状态。如果没有匹配的索引模板，数据流就无法滚动或创建后备索引。

    • 如果您的快照包含来自 App Search 或 Workplace Search 的数据，请确保您在恢复快照之前 已恢复企业搜索加密密钥。

    注意事项

    从快照恢复数据时，请记住以下几点：

    • 如果恢复数据流，也会恢复其支持索引。
    • 仅当现有索引已关闭并且快照中的索引具有相同数量的主分片时， 您才能恢复该索引。
    • 您无法恢复现有的打开索引。这包括数据流的支持索引。
    • 恢复操作会自动打开已恢复的索引，包括支持索引。

    获取可用快照的列表

    可以使用获取存储库 API和 获取快照 API来查找可用于恢复的快照。

    使用获取存储库 API 获取已注册快照存储库的列表。

    GET _snapshot

curl -X GET "localhost:9200/_snapshot?pretty"

    使用获取快照 API 获取特定存储库中的快照列表。这还会返回每个快照的内容。

    GET _snapshot/backup/*?verbose=false

curl -X GET "localhost:9200/_snapshot/backup/*?verbose=false&pretty"

    恢复索引或数据流

    默认情况下，恢复快照会尝试恢复快照中的所有索引和数据流，包括系统索引和系统数据流。一般情况下只需从快照中恢复特定索引或数据流。

    还原快照时重命名

    如果想避免删除现有数据，可以重命名恢复的索引和数据流。

    下面的示例将在所有恢复的索引或数据流的名称前添加 restored-。

    POST _snapshot/backup/snapshot-1/_restore
{
  "indices": "nginx-access-20240401, nginx-access-20240402",
  "rename_pattern": "(.+)",
  "rename_replacement": "restored-$1"
}

curl -X POST "localhost:9200/_snapshot/backup/snapshot-1/_restore?pretty" -H 'Content-Type: application/json' -d'
{
  "indices": "nginx-access-20240401, nginx-access-20240402",
  "rename_pattern": "(.+)",
  "rename_replacement": "restored-$1"
}
'
