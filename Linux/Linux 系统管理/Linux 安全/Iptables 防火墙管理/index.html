    Iptables 防火墙管理

    1、iptables 基本概念

    netfilter/iptables 是集成在内核中的包过滤防火墙系统。该架构可以实现数据包过滤，网络地址转换以及数据包管理功能。linux 中防火墙分为两部分：netfilter 和 iptables。netfilter 位于内核空间，目前是 Linux 内核的组成部分。netfilter 可以对本机所有流入、流出、转发的数据包进行查看、修改、丢弃、拒绝等操作。netfilter 位于内核空间中，用户无法接触内核和修改内核，需要使用 iptables 或 Firewalld 等工具来进行管理。

    1、iptables 概述

    netfilter/iptables: IP 信息包过滤系统，实际上由 netfilter 和 iptables 两个组件组成

    netfilter/iptables 关系：

    • netfilter 也称为内核空间 < kernelspace >，是内核的一部分，由一些信息包过滤组成，这些表包含内核用来控制信息包过滤处理的规则集
    • iptables 是一种工具，也称为用户空间 < userspace >，它使插入、修改和除去信息包过滤表中的规则变得容易

    netfileter/iptables 后期简称 iptables，iptables 是基于内核的防火墙，功能非常强大，iptables 内置了 filter、nat、mangle、raw 四张表，所有规则配置后，立即生效。

    2、iptables 的四表

    • filter
    # 描述：
    主要用于数据包过滤。该表根据系统管理员预定义的一组规则过滤符合条件的数据包。对于防火墙而言，主要利用在 filter 表中指定的规则来实现对数据包的过滤。filter 表是默认的表。在 filter 表中只能允许对数据包进行接受，丢弃的操作，而无法对数据包进行更改。

# 规则链：
    INPUT
    OUTPUT
    FORWARD
    • nat
    # 描述：
    主要用于 NAT，可以实现一对一、一对多、多对多等 NAT 工作，iptables 就是使用该表实现共享上网的。

# 规则链：
    PREROUTING
    POSTROUTING
    OUTPUT
    • mangle
    # 描述：
    主要用于修改数据包的属性。如 TCP 报文的 6 个标志位。mangle 表主要用于对指定数据包进行更改。

# 规则链：
    INPUT           # 处理进入的数据包
    FORWARD         # 处理转发的数据包
    OUTPUT          # 处理本地生成的数据包
    POSTROUTING     # 修改即将出去的数据包
    PREROUTING      # 修改即将到来的数据包
    • raw
    # 描述：
    优先级最高，用于异常处理，可以对收到的数据包在连接跟踪前进行处理。一旦用户使用了 raw 表，在 raw 某个链上处理完后，将跳过 nat 表和 ip_conntrack 处理，即不再做地址转换和数据包的链接跟踪处理了。

# 规则链：
    PREROUTING
    OUTPUT

    3、iptables 的五链

    • input < 入站数据过滤 >: 匹配目标 IP 是本机的数据包
    • output < 出站数据过滤 >: 出口数据包，一般不在此链上做配置
    • forward < 转发数据过滤 >: 匹配流经本机的数据包
    • prerouting < 路由前过滤 >: 用来修改目的地址，用来做 DNAT。如：把内网 80 端口映射到路由器外网端口上
    • postrouting < 路由后过滤 >: 用来修改源地址，用来做 SNAT。如：内网通过路由器 NAT 转换功能实现内网 PC 机通过一个公网 IP 地址上网

    2、iptables 基本原理

    1、处理数据包的原理

    整体数据包分类:

    • 发给防火墙本身的数据包

    • 经过防火墙的数据包

    当一个数据包进入网卡时，它首先进入 PREROUTING 链，内核根据数据包目的 IP 判断是否需要转送出去。如果数据包就是进入本机的，它就会沿着图向上移动，到达 INPUT 链。数据包到了 INPUT 链后，任何进程都会收到它。本机上运行的程序可以发送数据包，这些数据包会经过 OUTPUT 链，然后到达 POSTROUTING 链流出。

如果数据包是要转发出去的，且内核允许转发，数据包就会如图所示向右移动，经过 FORWARD 链，然后到达 POSTROUTING 链流出。

    2、表、链、规则的匹配顺序

    表间的优先顺序：raw --> mangle --> nat --> filter

    链间的优先顺序：

    • 入站数据：prerouting、input
    • 出站数据：output、postrouting
    • 转发数据：prerouting、forward、postrouting

    链内的规则匹配顺序：

    • 自上向下按顺序一次检查，找到匹配的规则即停止
    • 若在该链内长不到匹配的规则，则按该链的默认规则处理

    注意：规则的次序非常关键，谁的规则越严格，应该放的越靠前，而检查规则的时候，是按照从上往下的方式进行检查的。

    3、iptables 使用

    0、注意事项

    不指定表名时，默认表示 filter 表 不指定链名时，默认表示该表中所有链 除非设置了链的默认策略，否则需要指定匹配条件

    若规则列表中有多条相同的规则时，按内容匹配只删除规则号最小的一条 按规则号匹配删除时，规则号要 <= 已有规则数 按内容匹配删除时，内容要存在

    表名必须小写 链名必须大写 动作必须大写 模块必须小写

    1、命令

    1、语法

    iptables [-t table] OPTION [chain] 匹配条件 [-j 动作]

    2、OPTION

    • -N, --new chain # 创建自定义规则链
    # Example：

# 在 filter 表中创建 ZHUMS 链
iptables -N ZHUMS
    • -A, --append chain # 在链中追加规则。
    # Example:

# 在 filer 表 INPUT 链中追加拒绝并丢弃源地址为 192.168.100.151 目标地址为本机 IP 的数据包的规则
iptables -A INPUT -s 192.168.100.151 -j DROP

    3、匹配条件

    匹配条件包括通用匹配条件和扩展匹配条件。

    通用匹配条件是指针对源地址、目标地址的匹配，包括单一源 IP、单一源端口、单一目标 IP、单一目标端口、数据包流经的网卡以及协议。

    扩展匹配条件指通用匹配之外的匹配条件。

    1、通用匹配条件

    [!] -i, --in-interface input name[+] # 匹配入站网卡接口

    # Example:

# 拒绝并丢弃从 eth0 接口进来且源地址为 192.168.100.0/24 网段的数据包
iptables -A INPUT -i eth0 -s 192.168.100.0/24 -j DROP

    [!] -o, --out-interface output name[+] # 匹配出站网卡接口

    # Example:

# 拒绝并丢弃要从 eth1 接口发出且目标地址为 192.168.2.152 的数据包
iptables -A OUTPUT -o eth1 -d 192.168.2.152 -j DROP

    [!] -s, --source address[/mask][...] # 匹配源地址

    # Example:

# 拒绝并丢弃源地址为 192.168.100.151 的数据包
iptables -A INPUT -s 192.168.100.151/32 -j DROP

    [!] -d, --destination address[/mask][...] # 匹配目标地址

    # Example:

# 拒绝并丢弃目标地址为 192.168.100.152 的数据包
iptables -A INPUT -s 192.168.100.152/32 -j DROP

    [!] -p, --protocol proto # 匹配协议

    # Example:

# 拒绝并丢弃协议为 icmp 且目标地址为 192.168.100.151 的数据包。如：ping 不通，ssh 连得上。
iptables -A INPUT -p icmp -d 192.168.100.151 -j DROP 

# 拒绝并丢弃协议不是 icmp 且目标地址为 192.168.100.151 的数据包。如：ping 的通，ssh 连不上。
iptables -A INPUT ! -p icmp -d 192.168.100.151 -j DROP
    2、扩展匹配条件

    扩展匹配分为：隐式扩展和显式扩展；扩展功能需要配合模块实现。

    • 隐式扩展：在使用 -p 选项指明了特定的协议时，无需再用 -m, --match match 选项指明扩展模块的扩展机制，不需要手动加载扩展模块，即有的模块只需要指定协议名称，不需要指定模块名称，这样在定义协议时，间接的使用了模块，如 TCP、UDP、icmp 协议，因为协议名与模块名同名；
    • 显示扩展：必须使用 -m, --match match 选项指明要调用的扩展模块的扩展机制(手动指定引用的模块名称)，要手动加载扩展模块

    可通过 rpm -ql iptables | grep "[[:lower:]]\+\.so$" 命令查看可用的扩展模块

    可通过 iptables -m Modle_Name -h 命令或 iptables-extensions(8) 查看扩展模块的帮助信息

    1、隐式扩展匹配

    TCP 模块：

    [!] --sport, --source-port port[:port] # 匹配源端口；可以给出多个端口，但只能是连续的端口范围

    # Example:

# 拒绝并丢弃 192.168.100.152 通过 tcp 协议访问本机的 22 端口
iptables -A INPUT -s 192.168.100.152 -p tcp --dport 22 -j DROP

# 允许 192.168.100.152 通过 tcp 协议访问本机的 8080-8090 间的端口
iptables -A INPUT -s 192.168.100.152 -p tcp --dport 8080:8090 -j ACCEPT

    [!] --dport, --destination-port port[:port] # 匹配目标端口；可以给出多个端口，但只能是连续的端口范围

    [!] --tcp-flags mask comp # 匹配报文中 tcp 协议的标志位；Flags are: SYN ACK FIN RST URG PSH ALL NONE； 

    mask        # 要检查的 FLAGS list，以逗号分隔；
comp        # 在 mask 给定的诸多的 FLAGS 中，其值必须为 1 的 FLAGS 列表，余下的其值必须为 0；

    # filter表input链添加规则，指定 192.168.39.100 三次捂手的第一次握手直接拒绝；指定6个标记位，三次握手的第一次握手只有SYN标记位为1，其余都为0，所以最后的SYN表示为SYN为1；标记位只属于TCP协议，所以需要指定协议为TCP；第二次握手为SYN,ACK为1，其余为0；第三次握手为ACK为1，其余为0；并且标记位不存在6个都为1或者都为0的报文
iptables -A INPUT -s 192.168.39.100 -p tcp --tcp-flags SYN,ACK,FIN,RST,URG,PSH SYN -j REJECT

#--syn也表示三次握手的第一次握手，即只检查6个标记位中的4个，并且syn位为1
iptables -A INPUT -s 192.168.39.100 -p tcp --syn -j REJECT

    UDP 模块:

    [!] --sport, --source-port port[:port] # 匹配源端口；可以给出多个端口，但只能是连续的端口范围

    [!] --dport, --destination-port port[:port] # 匹配目标端口；可以给出多个端口，但只能是连续的端口范围

    ICMP 模块:

    [!] --icmp-type {type[/code]|typename}

    # ICMP 中 type 为 0 表示响应报文，为 8 表示请求报文。

    # Example:

# 允许 icmp 协议的响应报文
iptables -A INPUT -p icmp --icmp-type 0 -j ACCEPT
    2、显式扩展匹配条件

    multiport 模块：多端口匹配，最多15个

    [!] --sports, --source-ports port[,port|,port:port]... # 匹配多个源端口

    # Example:

# 拒接并丢弃源地址为 192.168.100.152 访问本机 tcp：22,80 端口的数据包
iptables -A INPUT -s 192.168.100.152 -p tcp -m multiport --dports 22,80 -j DROP

    [!] --dports, --destination-ports port[,port|,port:port]... # 匹配多个目标端口

    iprange 模块：

    [!] --src-range from[-to] # 匹配源地址范围

    iptables -I INPUT -d 172.16.0.7 -p tcp -m multiport --dports 22,80,139,445,3306 -m iprange --src-range 172.16.0.61-172.16.0.70 -j REJECT

    [!] --dst-range from[-to] # 匹配目标地址范围

    time 模块: 匹配数据包到达的时间

    --timestart hh:mm[:ss]

    --timestop hh:mm[:ss]

    [!] --weekdays day[,day...]

    [!] --monthdays day[,day...]

    --datestart YYYY[-MM[-DD[Thh[:mm[:ss]]]]]

    --datestop YYYY[-MM[-DD[Thh[:mm[:ss]]]]]

    --kerneltz # 使用内核配置的时区而非默认的UTC；

    string 模块： 匹配数据包中的字符

    --algo {bm|kmp}

    [!] --string pattern

    [!] --hex-string pattern

    --from offset

    --to offset

    iptables -I OUTPUT -m string --algo bm --string "gay" -j REJECT

    connlimit 模块：用于限制同一 IP 可建立的连接数

    --connlimit-upto n # 匹配连接数 <= n

    --connlimit-above n # 匹配连接数 >= n

    # Example:

# 当源地址 192.168.100.106 对本机 tcp:22 端口建立的连接数超过 3 个时，拒绝并丢弃数据包。
iptables -A INPUT -s 192.168.100.106 -p tcp --dport 22 -m connlimit --connlimit-above 3 -j DROP

    limit 模块：匹配收发数据包的速率

    --limit rate[/second|/minute|/hour|/day]

    --limit-burst number

    # Example:

# 允许目标地址是 10.0.0.166 转发速率为 50/s 的数据包;50/s 表示 1 秒中转发 50 个数据包
iptables -A FORWARD -d 10.0.0.166 -m limit --limit 50/s -j ACCEPT

    state 模块：限制收发包的状态

    [!] --state state # 匹配包的状态

    # 状态：NEW、RELATED、ESTABLISHED、INVALID
# NEW
    有别于 tcp 的 syn，如果我们发送一个流的初始化包，状态就会在 OUTPUT 链里被设置为NEW，当我们收到回应的包时，状态就会在 PREROUTING 链里被设置为ESTABLISHED 如果第一个包不是本地产生的，那就会在 PREROUTING 链里被设置为NEW状 态。
# RELATED
    连接态
# ESTABLISHED
    衍生态，与 conntrack 关联（FTP）
# INVALID
    不能被识别属于哪个连接或没有任何状态，这些状态可以一起使用，以便匹配数据包。这可以使我们的防火墙非常强壮和有效。以前，我们经常打开 1024 以上的所有端口来放行应答的数据。现在，有了状态机制，就不需再这样了。因为我们可以只开放那些有应答数据的端口，其他的都可以关闭。这样就安全多了。
    State < 状态 > Explanation < 注释 >
    NEW NEW说明这个包是我们看到的第一个包。意思就是，这是 conntrack 模块看到的某个连 接第一个包，它即将被匹配了。比如，我们看到一个 SYN 包，是我们所留意的连接的第一个 包，就要匹配它。第一个包也可能不是 SYN包，但它仍会被认为是NEW状态。这样做有时 会导致一些问题，但对某些情况是有非常大的帮助的。例如，在我们想恢复某条从其他的防火 墙丢失的连接时，或者某个连接已经超时，但实际上并未关闭时。
    ESTABLISHED ESTABLISHED 已经注意到两个方向上的数据传输，而且会继续匹配这个连接的包。处于 ESTABLISHED 状态的连接是非常容易理解的。只要发送并接到应答，连接就是ESTABLISHED 的了。一个连接要从NEW变 为 ESTABLISHED，只需要接到应答包即可， 不管这个包是发往防火墙的，还是要由防 火墙转发的。ICMP的错误和重定向等信息包也被 看作是 ESTABLISHED，只要它们是我们所发出的信息的应答。
    RELATED RELATED是个比较麻烦的状态。当一 个连接和某个已处于EST
    ABLISHED 状态的连接有关系时，就被认为是RELATED的了。换句话说，一个连接要想是RELATED的，首先要有一个 ESTABLISHED 的连接。这个 ESTABLISHED连接再产生一个主连接之外的连接，这个新的连接就是RELATED的了，当然前提是 conntrack 模块要能理解RELATED。ftp 是个很好的例子，FTP-data 连接就是和 FTP-control 有RELATED的。还有其他的例子，比 如，通过 IRC 的DCC 连接。有了这个状态，ICMP应 答、FTP 传输、DCC 等才能穿过防火 墙正常工作。注意，大部分还有一些UDP 协议都依赖这个机制。这些协议 是很复杂的，它们 把连接信息放在数据包里，并且要求这些信息能被正确理解。
    INVALID INVALID 说明数据包不能被识别属于哪个连接或没有任何状态。有几个原因可以产生这 种情况，比如，内存溢出，收到不知属于哪个连接的 ICMP 错误信息。一般地，我们DROP 这个状态的任何东西。

    mac 模块:

    [!]--mac-source # 匹配源 MAC地址

    报文经过路由后，数据包中原有的 mac 信息会被替换，所以在路由后的 iptables 中使用 mac 模块是没有意义的

    iptables -A INPUT -m mac --mac-source xx:xx:xx:xx:xx:xx -j DROP
    3、动作
    • ACCEPT # 放行数据包
    # 进行完此处理动作后，将不再比对其它规则，直接跳往下一个规则链。
    • DROP # 丢弃数据包
    # 进行完此处理动作后，将不再比对其它规则，直接中断过滤程序。
    • REJECT # 拒绝数据包
    # 拒绝该数据包，并传送数据包通知对方，可以传送的数据包有几个选择：ICMP port-unreachable、ICMP echo-reply 或是tcp-reset（这个数据包会要求对方关闭联机），进行完此处理动作后，将不再比对其它规则，直接 中断过滤程序。

    iptables -A FORWARD -p TCP --dport 22 -j REJECT --reject-with tcp-reset
    • SNAT # 源地址转换
    # 改写封包来源 IP 为某特定 IP 或 IP 范围，可以指定 port 对应的范围，进行完此处理动作后，将直接跳往下一个规则（mangleostrouting）。

    iptables -t nat -A POSTROUTING -p tcp-o eth0 -j SNAT --to-source 194.236.50.155-194.236.50.160:1024-32000
    • DNAT # 目的地址转换
    # 改写封包目的 IP 为某特定 IP 或 IP 范围，可以指定 port 对应的范围，进行完此处理动作后，将会直接跳往下一个规则（filter:input 或 filter:forward）。

    iptables -t nat -A PREROUTING -p tcp -d 15.45.23.67 --dport 80 -j DNAT --to-destination 192.168.1.1-192.168.1.10:80-100
    • LOG # 记录曰志
    # 将包信息记录在 /var/log 中，详细位置查看 /etc/syslog.conf 配置文件，进行完此处理动作后，将会继续比对其规则。

    iptables -A INPUT -p tcp -j LOG --log-prefix "INPUT packets"
    • MASQUERADE #　地址欺骗，动态源地址转换
    # 改写数据包来源 IP 为防火墙 NIC IP，可以指定 port 对应的范围，进行完此处理动作后，直接跳往下一个规则。这个功能与 SNAT 略有不同，当进行 IP 伪装时，不需指定要伪装成哪个 IP，IP 会从网卡直接读取，当使用拨号连接时，IP 通常是由 ISP 公司的 DHCP 服务器指派的，这个时候 MASQUERADE 特别有用。

    iptables -t nat -A POSTROUTING -p TCP -j MASQUERADE --to-ports 1024-31000
    • REDIRECT # 重定向
    # 将包重新导向到另一个端口（PNAT），进行完此处理动作后，将会继续比对其它规则。 这个功能可以用来实作通透式porxy 或用来保护 web 服务器。

    iptables -t nat -A PREROUTING -p tcp --dport 80 -j REDIRECT --to-ports 8080
    • MIRROR # 镜像数据包
    # 将来源 IP 与目的地 IP 对调后，将数据包送回，进行完此处理动作后，将会中断过滤程序。
    • QUEUE # 中断过滤程序
    中断过滤程序，将数据包放入队列，交给其它程序处理。透过自行开发的处理程序，可以进行其它应用，例如：计算联机费.......等。
    • RETURN 
    # 结束在目前规则链中的过滤程序，返回主规则链继续过滤，如果把自订规则链看成是一个子程序，那么这个动作，就相当提早结束子程序并返回到主程序中。
    • MARK
    # 将数据包标上某个代号，以便提供作为后续过滤的条件判断依据，进行完此处理动作后，将会继续比对其它规则。

    iptables -t mangle -A PREROUTING -p tcp --dport 22 -j MARK --set-mark 2

    2、自定义链

    注意：自定义链并不能直接使用，而是需要被默认链引用才能够使用

    1、创建自定义链

    iptables [-t table] -N | --new chain

    # 在 filter 表中创建 SSH_INPUT 链

iptables -t filter -N SSH_INPUT

    2、默认链引用自定义链

    # 将访问 tcp:22 端口的数据包交给 SSH_INPUT 链处理

iptables -I INPUT -p tcp --dport 22 -j SSH_INPUT

    3、给自定义链添加规则

    # 允许 192.168.100.0/24 网段数据包通过 22 端口；拒绝 192.168.100.152 数据包通过 22 端口。

iptables -I SSH_INPUT -s 192.168.100.0/24 -p tcp --dport 22 -j ACCEPT
iptables -I SSH_INPUT -s 192.168.100.152/32 -p tcp --dport 22 -j DROP

    4、删除自定义链

    [root@zhums ~]# iptables -X SSH
iptables: Too many links.           # SSH 链引用计数不为 0，被 INPUT 链引用了。
[root@zhums ~]# iptables -D INPUT 1
[root@zhums ~]# iptables -X SSH
iptables: Directory not empty.      # SSH 链规则不为空
[root@zhums ~]# iptables -D SSH 1

    修改链名后，默认链引用的 target 会自动同步。

    3、规则的备份与恢复

    1、备份

    Use: iptables-save [option]
     ip6tables-save [option]
Desc: 将 iptables 规则打印到 STDOUT
Option:
  -c, --counters                    # 在输出中包含所有数据包和字节计数器的当前值
  -t, --table tablename             # 指定打印的表，默认打印所有可用的表
  -M,--modprobe modprobe_program    # 指定 modprobe 程序的路径。默认情况下，iptables-save 将检查 /proc/sys/kernel/modprobe 以确定可执行文件的路径。

    iptables-save > /root/iptables.bak

    2、恢复

    Use: iptables-restore [option]
     ip6tables-restore [option]
Desc: 从 STDIN 上指定的数据恢复 iptables 规则
Option:
  -c, --counters            # 恢复所有数据包和字节计数器的值
  -n, --noflush             # 不刷新现有的 iptables 数据。默认会刷新（删除）相应表的所有数据
  -t, --test                # 只分析和构造规则集，但不提交它。
  -T, --table name          # 仅恢复指定的表规则，及时 STDIN 中包含其它表的数据
  -v, --verbose             # 打印调试信息
  -M, --modprobe modprobe_program       # 指定 modprobe 程序的路径。默认情况下，iptables restore 将检查 /proc/sys/kernel/modprobe 以确定可执行文件的路径。
  -w, --wait [seconds]  # 等待 xtables 锁。为了防止程序的多个实例同时运行，将尝试在启动时获得独占锁。默认情况下，如果无法获得锁。此选项将使程序等待（无限期或可选秒），直到获得独占锁定
  -W, --wait-interval microseconds  # 每次迭代等待的时间间隔。当运行延迟敏感的应用程序时，等待 xtables 锁的时间延长可能是不可接受的。此选项将使每次迭代都采用指定的时间量。默认间隔为 1 秒。此选项仅适用于 -w

    iptables-restore < /root/iptables.bak
