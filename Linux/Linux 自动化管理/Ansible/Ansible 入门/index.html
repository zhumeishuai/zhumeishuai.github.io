    Ansible 详解

    一、Ansible 简介

    1.1、Ansible 介绍

    • ansible 是目前最受运维欢迎的自动化运维工具，基于 Python 开发，集合了众多运维工具（SaltStack puppet、chef、func、fabric）的优点，实现了批量系统配置、批量程序部署、批量运行命令等功能。

    • ansible 是基于 paramiko 开发的,并且基于模块化工作，本身没有批量部署的能力。真正具有批量部署的是 ansible 所运行的模块，ansible 只是提供一种框架。ansible 不需要在远程主机上安装 client/agents，因为它们是基于 ssh 来和远程主机通讯的。ansible 目前已经已经被红帽官方收购，是自动化运维工具中大家认可度最高的，并且上手容易，学习简单。是每位运维工程师必须掌握的技能之一。

    1.2、Ansible 特点

    • 部署简单，只需在主控端部署 Ansible，被控端无需任何操作；
    • 默认使用 SSH 协议对设备进行管理；
    • 有大量常规运维操作模块，可实现日常绝大部分操作；
    • 配置简单、功能强大、扩展性强；
    • 支持 API 及自定义模块，可通过 Python 轻松扩展；
    • 通过 Playbooks 来定制强大的配置、状态管理；
    • 轻量级，无需在客户端安装 agent，更新时，只需在操作机上进行一次更新即可；
    • 提供一个功能强大、操作性强的 Web 管理界面和 REST API 接口 —— AWX 平台。

    1.3、Ansible 架构图

    # 组件说明

- Ansible： Ansible 核心程序。
- HostInventory： 定义 Ansible 管理的主机清单，包括端口、密码、ip、用户名等信息。
- Playbooks： "剧本" YAML 格式文件，多个任务定义在一个文件中，定义主机需要调用哪些模块来完成的功能。
- CoreModules： 核心模块，主要操作是通过调用核心模块来完成管理任务。
- CustomModules： 自定义模块，完成核心模块无法完成的功能，支持多种语言。
- Plugins： 插件，完成模块功能的补充，包括连接插件、邮件插件等 
- ConnectionPlugins： 连接插件，用于链接 Ansible 和 Host。

# 最重要的一点是 ansible 是模块化的 它所有的操作都依赖于模块

    二、Ansible 任务执行

    2.1、Ansible 任务执行模式

    • Ansible 系统由控制主机对被管节点的操作方式可分为两类，即 ad-hoc 和 playbook：
      • ad-hoc 模式 ( 点对点模式 )
        • 使用单个模块，支持批量执行单条命令。ad-hoc 命令是一种可以快速输入的命令，而且不需要保存起来的命令。就相当于 bash 中的一句话 shell。
      • playbook 模式 ( 剧本模式 )
        • 是 Ansible 主要管理方式，也是 Ansible 功能强大的关键所在。playbook 通过多个 task 集合完成一类功能，如 Web 服务的安装部署、数据库服务器的批量备份等。可以简单地把 playbook 理解为通过组合多条 ad-hoc 操作的配置文件。

    2.2、Ansible 命令执行流程

    1、读取配置文件，默认 "/etc/ansible/ansible.cfg";
2、读取对应的主机的配置文件，找到要执行的主机或者主机组;
3、加载对应的模块文件，如 "command";
4、将模块生成对应的临时 py 文件传输至目标主机，对应执行用户的家目录的 ".ansible/tmp/XXX/XXX.PY" 文件;
5、给文件增加执行权限;
6、执行并返回结果;
7、删除临时 py 文件，sleep 0 退出;

    2.3、Ansible 执行返回的颜色说明

    黄色      # 对远程节点进行相应修改
绿色      # 对远程节点不进行相应修改，或者只是对远程节点信息进行查看
红色      # 操作执行命令有异常
紫色      # 表示对命令执行发出警告信息（可能存在的问题，给你一下建议）

    yum install -y epel-release

    • 安装目录如下( yum 安装 )：
      • 配置文件目录：/etc/ansible/
      • 执行文件目录：/usr/bin/
      • Lib 库依赖目录：/usr/lib/pythonX.X/site-packages/ansible
      • Help 文档目录：/usr/share/doc/ansible-X.X.X/　　
      • Man 文档目录：/usr/share/man/man1/

    3.5、Ansible 配置文件的优先级

    1. $ANSIBLE_CONFIG: 检查环境变量 ANSIBLE_CONFIG 指向的配置文件
    2. ./ansible.cfg: 检查当前目录下的 ansible.cfg 配置文件
    3. ~/.ansible.cfg: 检查当前用户家目录下的 .ansible.cfg 配置文件
    4. /etc/ansible/ansible.cfg: 检查 /etc/ansible/ 目录下的 ansible.cfg 配置文件

    3.6、Ansible 配置文件

    3.6.1、字段说明

    [defaults]                  # 通用配置项

[selinux]                   # selinux 相关配置项
[colors]                    # ansible 命令输出的颜色相关配置项
[diff]                      # 定义是否在运行时打印 diff 即变更前后的差异

    3.6.2、常用配置项

    • [defaults]
    inventory=/etc/ansible/hosts        # 主机清单 inventory 文件的位置
library=/usr/share/ansible          # 指向存放 Ansible 模块的目录，支持多个目录方式，只要用冒号（ : ）隔开就可以
forks=5                             # 并发连接数，默认为 5
remote_user=ansible                 # 指定默认使用什么用户连接远程主机
remote_port=22                      # 指定连接被管节点的管理端口，默认为 22 端口，建议修改，能够更加安全
sudo_user=root                      # 指定 sudo 使用的默认用户
host_key_checking=False             # 是否检查 SSH 主机的密钥
ask_pass=false                      # 是否使用密码认证
timeout=60                          # SSH 连接超时时间，单位为秒
log_path=/var/log/ansible.log       # 指定 ansible 日志存储位置（默认不记录日志 ）
    • [privilege_escalation]
    become=True             # 若使用远程主机的普通用户，该用户是否允许提权
become_method=sudo      # 若允许提权，使用哪种方式提权，默认 sudo
become_user=root        # 提权到的用户，默认 root
become_ask_pass=False   # 提权时，是否需要密码验证，默认 False
    • [ssh_connection]
    ssh_args=               # ansible 通过 ssh 连接远程主机，用于定义 ssh 连接时的参数
host_key_checking=      # 是否检查 SSH 主机的密钥，或在 ssh_args 中添加参数 -o StrictHostKeyChecking=no

    [defaults]
inventory=/opt/hosts

[privilege_escalation]

    5.16、lineinfile 模块

    1、lineinfile 模块介绍

    • 用来替换某一段文本，或删除某一段文本

    2、lineinfile 模块选项

    path            # 指定要操作的文件。
regexp          # 使用正则表达式匹配要修改的行，当替换文本时，如果多行被匹配，则只有最后被匹配到的行被替换，当删除文本时，如果多行被匹配，这些行都会被删除。
line            # 修改或插入行的内容

insertbefore    # 在匹配行之前插入，insertbefore 参数的值可以设置为 BOF 或者正则表达式，BOF 为 Begin Of File 之意，表示插入到文档的开头，如果将 insertbefore 的值设置为正则表达式，表示将文本插入到匹配到正则的行之前，如果正则没有匹配到任何行，则插入到文件末尾，当使用 backrefs 参数时，此参数会被忽略。
insertafter     # 在匹配行后插入，insertafter 参数的值可以设置为 EOF 或者正则表达式，EOF 为 End Of File 之意，表示插入到文档的末尾，默认情况下 insertafter 的值为 EOF，如果将 insertafter 的值设置为正则表达式，表示将文本插入到匹配到正则的行之后，如果正则没有匹配到任何行，则插入到文件末尾，当使用 backrefs 参数时，此参数会被忽略。

backup          # 是否在修改文件之前对进行备份。
create          # 文件不存在时，是否创建对应的文件。
validate        # 通过文件自带的验证机制验证文件修改的有效性。
state
    present     # 不删除，默认值
    absent      # 删除
backrefs        # 默认情况下，当根据正则替换文本时，即使 regexp 参数中的正则存在分组，在 line 参数中也不能对正则中的分组进行引用，除非将 backrefs 参数的值设置为 yes。backrefs=yes 表示开启后向引用，这样，line 参数中就能对 regexp 参数中的分组进行后向引用了，这样说不太容易明白，可以参考后面的示例命令理解。backrefs=yes 除了能够开启后向引用功能，还有另一个作用，默认情况下，当使用正则表达式替换对应行时，如果正则没有匹配到任何的行，那么 line 对应的内容会被插入到文本的末尾，不过，如果使用了 backrefs=yes，情况就不一样了，当使用正则表达式替换对应行时，同时设置了 backrefs=yes，那么当正则没有匹配到任何的行时，则不会对文件进行任何操作，相当于保持原文件不变。

    六、Playbook 详解

    6.1、Playbook 简介

    • Playbook 是 ansible 用于配置、部署和管理被控节点的剧本。

    • 通过 Playbook 的详细描述，执行其中的一系列 tasks ，可以让远端主机达到预期的状态。playbook 就像 Ansible 控制器给被控节点列出的的一系列 to-do-list ，而被控节点必须要完成。

    • Playbook 字面意思，即剧本，现实中由演员按照剧本表演，在 Ansible中，这次由计算机进行表演，由计算机安装，部署应用，提供对外服务，以及组织计算机处理各种各样的事情。

    6.2、Playbook 使用场景

    • 执行一些简单的任务，使用 ad-hoc 命令可以方便的解决问题，但是有时一个设施过于复杂，需要大量的操作时候，执行的 ad-hoc 命令是不适合的，这时最好使用 playbook。
    • 就像执行 shell 命令与写 shell 脚本一样，也可以理解为批处理任务。
    • 使用 playbook 你可以方便的重用这些代码，可以移植到不同的机器上面，像函数一样，最大化的利用代码。可以把常见的应用都编写成 playbook，之后管理服务器会变得十分简单。

    6.3、Playbook 格式简介

    Playbook 使用 Yaml 语言编写。Yaml 语言是专门用来编写配置文件的语言，非常的简洁和强大。

    Yaml 的基本语法规则：

    • 文件的第一行应该以 "---" 开始，表明 YMAL 文件的开始

    • 大小写敏感

    • 使用缩进表示层级关系
    • 缩进只允许使用空格
    • 对于缩进的数量不敏感，只要相同层级的元素左对齐即可
    • # 表示行注释

    • 黄色: 系统状态发生改变
    • 红色: 执行失败，显示错误输出　

    6.6、Playbook 组成结构

    Playbook 由一个或多个 "play" 组成。"play" 主要对一组主机应用 "play" 中定义好的 task。从根本上说 task 就是对 ansible 模块的调用。而将多个 play 按照一定顺序组织到一个 playbook 中，称之为编排。

    Playbook 由以下四部分组成:

    • Target: 用于定义要执行 Playbook 的远程主机用户组、连接远程主机的方式等
    • Variables: 定义 Playbook 执行需要的变量
    • Tasks: 定义要执行的任务列表
    • Handlers: 定义 task 执行完后需要调用的任务

    Target 部分

    用于定义要执行 Playbook 的远程主机用户组、连接远程主机的方式等

    name            # 该 Play 的描述信息
hosts           # 指定在哪些主机上进行操作
remote_user     # 指定使用什么用户来操作远程主机
gather_facts    # 指定在执行 tasks 前是否先使用 setup 模块获取主机信息
force_handlers  # 指定是否强制执行被触发 handlers 任务；yes/no

    Variables 部分

    定义 Playbook 执行需要的变量

    vars            # 定义 tasks 中要使用的变量

    Tasks 部分

    tasks           # 具体要执行任务
  - name            # 该任务的描述信息
    MODULE          # 要执行的模块
    ignore_errors   # 是否忽略当前 task 运行发生的错误

    Handlers 部分

    • 在 Playbook 中 handler 事实上也是个 task，不过这个 task 默认并不执行，只有在被触发时才执行。
    • handler 通过 notify 来监视某个或多个 task，一旦 task 执行结果发生变化，即运行后为 "changed" 则触发 handler 执行响应操作。
    • handler 会在所有 play 都执行完毕后才会执行，这样可以避免当 handler 监视多个 task 执行结果都发生变化之后导致 handler 的重复执行。
    • 任务可以通过 "tags" 打标签，而后可在 ansible-playbook 命令上使用 -t 指定进行调用；

    3、注册变量

    1、注册变量介绍
    • 有时候需要将某一条任务执行的结果保存下来，以便在接下来的任务中调用。
    • 可以通过 register 关键字将某一任务结果保存为一个变量
    2、注册变量示例
    - name: registeer variables
  hosts: all
  tasks:
  - name: capture output of whoami command
    shell: whoami
    register: login
  - debug:
      var: login

    # debug 输出说明
"login": {      # 变量名，值为一个字典
    "changed": true,        # 状态是否发生了改变
    "cmd": "whoami",        # 调用的命令
    "delta": "0:00:00.003965",      # 任务执行的时间
    "end": "2023-03-14 19:27:21.727619",    # 任务结束的时间
    "failed": false,        # 任务是否运行失败
    "rc": 0,                # 返回值，0 表示正常，非 0 表示异常
    "start": "2023-03-14 19:27:21.723654",  # 任务开始的时间
    "stderr": "",           # 标准错误的输出，值为字符串
    "stderr_lines": [],     # 标准错误的输出，值为列表
    "stdout": "root",       # 标准输出，值为字符串
    "stdout_lines": [       # 标准输出，值为列表
        "root"
    ]
}

    注意: register 注册的变量的结果未必是一成不变的，在不确定返回值的情况下，尽量调试看看输出结果。

    • 在远端服务器获取目录下的文件，然后下载这些文件
    • 在 handler 执行前，发现前面一个 task 发生了 changed，然后执行一个指定的 task
    • 获取远端服务器 ssh key 的内容，构建出 known_hosts 文件

    4、使用 lookup 生成变量

    通常情况下，所有的配置信息都会被作为 ansible 的变量保存，而且可以保存在 ansible 允许定义变量的各种地方，如：vars 字段、vars_files 加载的文件中、host_vars 和 group_vars 目录中。

    有时候我们希望从文本文件或 .csv 文件中收集数据作为 ansible 的变量，或者直接获取某些命令的输出作为 ansible 的变量，这时候我们就需要通过 ansible 的 lookup 插件来从这些数据源中读取配置数据传递给 ansible 变量，并在 playbook 或模板中使用这些数据，注意：lookup 获取的变量来自于主控端。

    ansible 支持一套从不同数据源获取数据的 lookup，包括 file、password、pipe、env、template、csvfile、dnstxt、redis_kv、etcd 等。

    常用的 lookup 生成变量的方法

    1、file

    使用 file lookup 可以从文本文件中获取数据，并在这些数据传递给 ansible 变量，在 playbook 或 jinja2 模板中引用。

    • 获取 ssh 公钥并生成变量示例:
    - name: get ssh public_key from file module
  hosts: node2
  tasks:
  - name: set facts
    set_fact:
      public_key: "{{ lookup('file','/root/.ssh/id_rsa.pub') }}"
  - name: debug set facts
    debug:
      var: public_key
    2、pipe

    使用 pipe lookup 可以直接调用外部命令，并将命令执行的结果打印到标准输出，作为 ansible 变量。

    • 通过 pipe 调用 date 命令拿到一个以时间数字组成的字符串
    - name: get date command output
  hosts: node2
  tasks:
  - name: set facts
    set_fact:
      time_info: "{{ lookup('pipe', 'date +%Y%m%d%H%M%S') }}"
  - name: debug set facts
    debug:
      var: time_info
    3、env

    使用 env lookup 可以获取控制端上的环境变量

    • 获取控制端上的 $Host_Type 变量
    - name: get vars
  hosts: node2
  tasks:
  - name: get $Host_Type
    debug:
      msg: "{{ lookup('env', 'Host_Type') }}"

    5、魔法变量

    Ansible 默认提供了一些内置的变量以实现一些特定的功能，称之为魔法变量。

    1、hostvars

    获取某台指定主机的相关变量。

    • 如果有一台 web 服务器的配置文件需要指定 db 服务器的 ip 地址，我们假设这台 db 的 hostname 为 db.example.com，ip 地址绑定在 eth0 网卡上，我们可以通过如下方法在 web 服务器上调用 db 服务器的地址。
    - name: get vars
  hosts: node1
  tasks:
  - name: get node1 ipv4 address
    debug:
      msg: "{{ hostvars['node1'].ansible_eth0.ipv4.address }}"
    2、inventory_hostname

    是 ansible 所识别的当前正在运行 task 的主机的主机名

    • 获取当前运行 task 的主机的 ipv4 地址
    - name: get vars
  hosts: node1,node2
  tasks:
  - name: get $Host_Type
    debug:
      msg: "{{ hostvars[inventory_hostname].ansible_eth0.ipv4.address }}"
    3、groups

    groups 是 inventory 中所有主机组的列表，可用于枚举主机组中的所有主机。

    - name: get vars
  hosts: node1
  tasks:
  - name: get
    debug:
      msg:
        - "{{ groups }}"            # 列出所有的主机组
        - "{{ groups.webserver }}"  # 列出 dbserver 组中的所有主机
        - "{{ group_names }}"       # 列出当前正在执行 task 的目标主机位于的主机组

    6.9、Playbook 的判断

    1、when 语句

    • when 判断的对象是 task，所以和 task 在同一列表层次。它的判断结果决定它所在 task 是否执行，而不是它下面的 task 是否执行。
    • when 中引用变量的时候不需要加 {{ }} 符号。
    1、基本示例
    • 判断远程主机的发行版本是否为 RedHat
    - name: test system version
  hosts: node1
  tasks:
    - when: ansible_os_family == "RedHat"
      debug:
          msg: "Host is RedHat"

    2、Playbook 中的运算符

    1、比较运算符
    ==      # 比较两个对象是否相当，相等返回真。可用于比较字符串与数字
!=      # 比较两个对象是否不等，不等则返回真。
>       # 比较两个对象的大小，左边的值大于右边的值,则为真
<       # 比较两个对象的大小，左边的值小于右边的值,则为真
>=      # 比较两个对象的大小，左边的值大于等于右边的值,则为真
<=      # 比较两个对象的大小，左边的值小于等于右边的值,则为真
    • 判断远程主机的剩余内存是否大于等于 2000M
    - name: test system version
  hosts: node1
  tasks:
    - when: ansible_memfree_mb >= 2000
      debug:
          msg: "free memory >= 2000MB"
    2、逻辑运算符
    and     # 逻辑与，左边和右边的表达式同时为真，则返回真
or      # 逻辑或，左边和右边的表达式任意一个为真，则返回真
not     # 取反，对表达式的结果取反
()      # 一组表达式组合在一起，形成一个更大的表达式，组合内的所有表达式都是逻辑与的关系
    • 判断远程主机的发行版是否为 RedHat 且版本大于等于 8.0
    - name: test system version
  hosts: node1
  tasks:
    - when: ansible_distribution == 'RedHat' and ansible_distribution_version >= '8.0'                                                     
      debug:
          msg: "is RedHat and >= 8.0"
    • 判断远程主机的发行版是否大于等于 RedHat 8.4 或大于等于 CentOS 7.6
    - name: test system version
  hosts: node1
  tasks:
    - when: (ansible_distribution == 'RedHat' and ansible_distribution_version >= '8.0') or (ansible_distribution == 'CentOS' and ansible_distribution_version >= '7,6')                             
      debug:
          msg: "Test is True"
    • 判断 register 注册变量的返回结果
    - name: test system version
  hosts: node1
  tasks:
    - name: get httpd service status
      shell: "/usr/bin/systemctl is-active httpd"
      register: status
      ignore_errors: yes
    - name: restart service
      when: status.rc != 0                                                                                     
      systemd: name=httpd state=restarted

    4、block

    在 ansible 中，使用 block 将多个任务进行组合，当作一个整体。我们可以对这一个整体做条件判断，当条件成立时，则执行 block 中的所有任务。

    1、基本示例
    - hosts: node1
  tasks:
    - block:
        - file:
              path: /tmp/node1.txt
              state: touch
        - debug:
              msg: "debug info"
      when: ansible_os_family == "RedHat"
    2、rescue

    在 Ansible playbook 中，rescue关键字用于指定当一个 block中的任务失败时要执行的任务。

    - hosts: node1
  gather_facts: false
  tasks:
    - block:
        - shell: "ls /abc"
      rescue:
        - debug:
              msg: "/abc not exists"
    3、always

    在 Ansible playbook 中，always 关键字用于指定无论 block 中的任务是否失败，都要执行的任务。

    - hosts: node1 
  gather_facts: false
  tasks:
    - block:
        - shell: "ls /abc"
      rescue:
        - debug:
              msg: "block tasks run faild"
      always:
        - debug:
              msg: "Welcome to Linux"

    5、错误处理

    1、fail 模块
    2、failed_when

    failed_when是 Ansible playbook 中的一个条件。可以使用 failed_when 来定义任务失败的条件。当 failed_when 对应的条件成立时，Ansible 会将对应任务的执行状态设置为失败，以停止 playbook 的运行。

    • 检查命令是否返回非零退出状态，并在命令返回非零退出状态时停止 playbook:
    - hosts: node1
  gather_facts: false
  tasks:
    - shell: "ls /root/.bash_history"
      register: result
      failed_when: result.rc != 0

    八、Ansible role 角色管理

    8.1、role 角色管理

    在 Ansible 中 role 是将 playbook 分割为多个文件的主要机制。大大简化了复杂 playbook 的编写，同时使得他们非常易于复用。

    1、创建 role 的步骤

    1. 创建以 roles 命名的目录
    2. 在 roles 目录中分别创建角色名称命名的目录，如 webserver 等
    3. 在每个角色命名的目录中分别创建 files、handlers、meta、tasks、template 和 vars 目录，用不到目录也可以不创建
    4. 在 playbook 文件中调用该角色

    2、role 的基本构成

    • files: 存放任务中需要用到的文件，如 https 证书等。
    • template: 存放角色相关的 jinja2 模板文件，当使用角色相关的模板时，若未指定模板路径，则默认在该目录中查找。
    • tasks: 存放角色要执行的所有任务文件，包含一个主文件 main.yaml，可以在主文件中通过 include 的方式引用其它任务文件。
    • handlers: 存放定义角色中需要调用的 handlers，包含一个主配置文件 main.yaml，可通过 include 引入其它的 handlers 文件。
    • vars: 存放角色需要的变量，包含一个主文件 main.yaml。
    • meta: 存放角色的元数据信息，如角色的描述、作者、角色的作用、角色的依赖关系等。默认这些信息会写入到该目录中的 main.yaml 文件中。
    • defaults: 存储该角色用到的变量，与 vars 不同的是，defaults 中定义的变量的优先级是最低的。
    roles/
└── zhums
    ├── defaults
    │   └── main.yml
    ├── files
    ├── handlers
    │   └── main.yml
    ├── meta
    │   └── main.yml
    ├── README.md
    ├── tasks
    │   └── main.yml
    ├── templates
    ├── tests
    │   ├── inventory
    │   └── test.yml
    └── vars
        └── main.yml
